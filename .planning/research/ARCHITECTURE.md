## Component Boundaries

### 1. Content Source (`docs/`)
- **Boundary:** Markdown files with YAML front matter
- **Owns:** Chapter content, assets (images, CSS), downloadable resources
- **Interface:** `mkdocs.yml` references files by relative path; navigation defined in `mkdocs.yml`
- **External interface:** Consumed by `mkdocs build` command

### 2. Site Configuration (`mkdocs.yml`)
- **Owns:** Theme settings, navigation structure, markdown extensions, plugins, site metadata
- **Interface:** Directives consumed by `mkdocs` CLI during build
- **External interface:** Defines what gets built and how it appears

### 3. MkDocs Build Engine
- **Owns:** Transform of Markdown + config into static HTML site
- **Input:** `docs/` + `mkdocs.yml`
- **Output:** `site/` directory (static HTML, CSS, JS)
- **Interface:** `mkdocs build --strict` command (strict mode catches broken links/missing refs)
- **External interface:** Output directory consumed by deployment step

### 4. GitHub Actions Workflow (`.github/workflows/deploy.yml`)
- **Owns:** CI/CD pipeline -- triggered on push to `main`
- **Stages:**
  1. Checkout code
  2. Set up Python (for MkDocs)
  3. Install dependencies (`mkdocs`, `mkdocs-material`, plugins)
  4. Run `mkdocs build`
  5. Validate with `mkdocs build --strict`
  6. Deploy `site/` directory to Cloudflare Pages
- **Interface:** Uses `cloudflare/pages-action` for deployment
- **External interface:** Communicates with Cloudflare Pages API via secret credentials

### 5. Cloudflare Pages ( Hosting)
- **Owns:** Static file hosting, CDN distribution, custom domain, HTTPS
- **Interface:** Receives built `site/` directory from GitHub Actions
- **External interface:** Serves site at `https://[project].pages.dev` or custom domain

## Data Flow

```
[Author writes .md files]
         │
         ▼
[GitHub Actions trigger (push to main)]
         │
         ▼
[Checkout + Python setup]
         │
         ▼
[Install: mkdocs, mkdocs-material, plugins]
         │
         ▼
[mkdocs build --strict]
         │         │
         │    [site/ directory]
         │         │
         ▼         ▼
[Validation passes] ──► [cloudflare/pages-action deploy]
                                        │
                                        ▼
                              [Cloudflare Pages CDN]
                                        │
                                        ▼
                              [Public site served]
```

**Data direction:** Content flows one way only -- from markdown source to deployed static site. No database, no user-generated content, no authentication. This is a purely static publishing pipeline.

## Suggested Build Order (Phase Dependencies)

| Phase | Component | Why First / Why After |
|-------|-----------|-----------------------|
| 1 | Content (`docs/`) + `mkdocs.yml` | All publishing derives from these; MkDocs will fail without valid config |
| 2 | `mkdocs build` locally | Verify site renders correctly before automating; catch broken links early |
| 3 | GitHub Actions workflow | Once local build works, automate with CI; adds CI benefit (link validation, etc.) |
| 4 | Cloudflare Pages integration | Connect CI output to hosting; requires Cloudflare account + `CLOUDFLARE_API_TOKEN` secret |
| 5 | Custom domain + HTTPS | After verifying basic deployment works; Cloudflare handles certs automatically |
| 6 | EPUB build (`build.sh`) | Independent artifact generation; does not affect site deployment |

**Critical path:** Content + mkdocs.yml (phase 1) --> local build (phase 2) --> GitHub Actions (phase 3) --> Cloudflare Pages (phase 4)

## Cloudflare Pages Deployment

Cloudflare Pages accepts deployments in two ways relevant here:

1. **Git integration (recommended):** Connect repo to Cloudflare Pages; Pages auto-deploys on push. This can replace the GitHub Actions approach above, but offers less control over build commands.

2. **Direct upload via `wrangler` or `pages-action`:** GitHub Actions builds locally, then uploads the `site/` directory. Gives full control over build environment. **This is the recommended approach** because:
   - Full control over Python version and dependency installation
   - Consistent with MkDocs tooling (which is Python-based)
   - `mkdocs build --strict` for link validation runs in CI before any deployment happens

### Required Secrets (GitHub repository settings)

| Secret | Purpose |
|--------|---------|
| `CLOUDFLARE_API_TOKEN` | Authorizes GitHub Actions to deploy to Cloudflare Pages |
| `CLOUDFLARE_ACCOUNT_ID` | Identifies the Cloudflare account |

## Anti-Patterns to Avoid

### Anti-Pattern 1: Skipping Local Build Validation
**What goes wrong:** Pushing broken content triggers CI failure, wastes CI minutes, delays publication.
**Instead:** Always run `mkdocs build --strict` locally before pushing.

### Anti-Pattern 2: Cloudflare Git Integration + GitHub Actions Hybrid
**What goes wrong:** Two deployment triggers (Cloudflare Git integration AND GitHub Actions) race each other, causing doubled deployments or stale caches.
**Instead:** Choose one -- either use Cloudflare Git integration (no `pages-action` needed) OR use GitHub Actions with direct upload (disable Cloudflare Git integration).

### Anti-Pattern 3: Hardcoding `site_url` Without Understanding its Impact
**What goes wrong:** `site_url` in `mkdocs.yml` must match the deployed URL for sitemap and canonical URL generation to work correctly.
**Instead:** Set `site_url` to the actual deployed URL (or leave it unset for Cloudflare Pages auto-detection).

## Scalability Considerations

| Scale | Concern | Mitigation |
|-------|---------|------------|
| 10 chapters | Build time | MkDocs builds are fast (<30s); no action needed |
| 50 chapters | Asset management | Store images in `docs/assets/`; use relative paths |
| 100+ chapters | Navigation complexity | Use `navigation.sections` in `mkdocs.yml` to group chapters |
| Multiple contributors | Build consistency | GitHub Actions ensures every push builds identically |
| Custom domain | SEO + trust | Add custom domain in Cloudflare Pages settings; HTTPS is automatic |
